Why I'm Building in Public as a Junior Developer

Putting unfinished work on GitHub felt terrifying. Six months later, it's the best decision I've made for my career.

Six months ago, the idea of pushing my code to a public GitHub repository made me anxious. What if experienced developers saw my code and judged it? What if my variable names were wrong, my architecture was naive, or my commit messages were embarrassing?

I pushed it anyway. Here's why that was the right call.

The Fear Is Universal

Every developer I've spoken to — from juniors to seniors — felt this way at some point. The difference is that senior developers pushed through the discomfort years ago and now have years of public work to show for it.

The uncomfortable truth: nobody is scrutinising your code as harshly as you think. Most developers are too busy with their own work to audit a stranger's repository. And the ones who do look? They're usually impressed that you're building and sharing, not judging your implementation details.

What Building in Public Actually Looks Like

For me, it means:

  • Every project goes on GitHub, even if it's messy or incomplete
  • I write about what I learn, even if the topic feels "too basic" to share
  • I document my decisions in README files so future-me (and anyone else) can understand why I built things a certain way

It doesn't mean streaming myself coding or posting daily updates on social media. It's quieter than that — just making my work visible and my learning process transparent.

The Unexpected Benefits

It forces you to write better code

When you know someone might read your code, you naturally write cleaner code. You add comments. You structure your folders. You write a proper README. These habits compound over time and make you a better developer even when nobody is watching.

It creates a portfolio that speaks for itself

When I apply for jobs, I don't need to explain what I can build — my GitHub shows it. The staff rostering system demonstrates that I can build a full-stack application with authentication, a database, a constraint solver, and a polished frontend. That's more convincing than any bullet point on a CV.

It connects you with other developers

After publishing my first project, I got messages from other beginners asking how I implemented certain features. Those conversations taught me things I wouldn't have learned alone. The developer community is more supportive than you'd expect.

How to Start

If you're hesitating, here's my advice: start with something you've already built. It doesn't need to be perfect. Push it to GitHub, write a README that explains what it does and how to run it, and share it.

You'll feel exposed for about a day. Then you'll feel proud. And then you'll start building the next thing, knowing it'll be even better than the first.

The best time to start building in public was when you wrote your first line of code. The second best time is now.
