Deploying a Full-Stack App for the First Time — What I Wish I Knew

localhost worked perfectly. Production was a different story. Here's every gotcha I hit deploying a React + FastAPI + PostgreSQL app.

There's a particular kind of panic you feel when your app works flawlessly on localhost but breaks the moment you try to deploy it. I experienced this several times while getting my staff rostering system live. Here's everything I wish someone had told me.

The Gap Between Dev and Production

On my machine, the frontend runs on localhost:5173 and the backend on localhost:8000. They talk to each other just fine. In production, everything is different: different URLs, HTTPS instead of HTTP, CORS policies, environment variables, and database connection strings.

Nobody warns you about this gap in tutorials.

CORS Will Block You

The first thing that broke in production was CORS (Cross-Origin Resource Sharing). My React app on one domain couldn't call my FastAPI backend on another domain. The browser blocked every request with a cryptic error.

The fix in FastAPI is straightforward:

from fastapi.middleware.cors import CORSMiddleware

app.add_middleware(
    CORSMiddleware,
    allow_origins=["https://your-frontend-domain.com"],
    allow_credentials=True,
    allow_methods=["*"],
    allow_headers=["*"],
)

But here's the gotcha: don't use allow_origins=["*"] in production. It works, but it means any website can call your API. Be explicit about which domains you trust.

Environment Variables Are Not Optional

I hardcoded my database URL during development. In production, this meant my credentials were sitting in the source code on GitHub. Not ideal.

The fix: use environment variables for anything sensitive — database URLs, JWT secret keys, API keys. In FastAPI, I used python-dotenv:

import os
from dotenv import load_dotenv

load_dotenv()
DATABASE_URL = os.getenv("DATABASE_URL")
JWT_SECRET = os.getenv("JWT_SECRET")

And a .env file (added to .gitignore) for local development.

Database Migrations

On localhost, I'd just drop and recreate tables whenever I changed a schema. In production, you can't do that — real data lives there. This is where migration tools like Alembic become essential.

I learned this the hard way by losing test data on a staging server. Set up Alembic (or equivalent) from the start, even if it feels like overkill for a small project.

The Build Step Matters

React development mode and production mode behave differently. npm run dev uses hot reloading, verbose error messages, and unminified code. npm run build produces optimised, minified output.

I hit a bug that only appeared in the production build — a component worked in dev mode but failed after minification because of a naming conflict. Always test the production build locally before deploying:

npm run build
npm run preview

My Deployment Checklist

After several painful deployments, I now follow this checklist:

  1. All secrets are in environment variables, not in code
  2. CORS is configured for specific production domains
  3. Database migrations are up to date
  4. Production build runs locally without errors
  5. HTTPS is configured
  6. .env is in .gitignore

It's simple, but it would have saved me hours of debugging if I'd had it from the start.
