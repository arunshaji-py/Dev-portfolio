I chose PostgreSQL for my first full-stack project because it's the industry standard. I figured I'd pick it up as I went. That was partially true — I did learn, but mostly by making mistakes and then fixing them.

Here's what I wish I'd understood earlier.

## Design Your Tables Before Writing Code

My first attempt at a database schema had employees, shifts, and assignments all in one table. It was a mess — duplicate data everywhere, no way to cleanly query "which shifts are uncovered," and updates required changing data in multiple rows.

The fix was normalisation: separate tables for employees, shifts, wards, assignments, and availability, connected by foreign keys. Each piece of data lives in exactly one place.

```sql
CREATE TABLE assignments (
    id SERIAL PRIMARY KEY,
    employee_id INTEGER REFERENCES employees(id),
    shift_id INTEGER REFERENCES shifts(id),
    assigned_at TIMESTAMP DEFAULT NOW()
);
```

Foreign keys aren't just about data integrity — they force you to think about relationships between entities before you start coding.

## SQLAlchemy Makes Life Easier (Mostly)

Using SQLAlchemy as my ORM meant I could write Python instead of raw SQL for most operations. But the abstraction has a cost: you need to understand what SQL it's generating underneath.

I had a query that loaded all shifts with their assignments and employee details. In Python it looked clean:

```python
shifts = db.query(Shift).all()
for shift in shifts:
    print(shift.assignment.employee.name)
```

But this generated one SQL query per shift (the "N+1 query problem"). With 200 shifts, that's 200 separate database calls. Adding `joinedload` fixed it:

```python
from sqlalchemy.orm import joinedload

shifts = db.query(Shift).options(
    joinedload(Shift.assignment).joinedload(Assignment.employee)
).all()
```

One query instead of 200. The page went from 8 seconds to under 100ms.

## Indexes Are Not Premature Optimisation

I initially thought indexes were something you worry about at scale. But even with a few thousand rows, queries filtering by `employee_id` or `shift_date` were noticeably slow without them.

Adding an index is one line:

```sql
CREATE INDEX idx_shifts_date ON shifts(date);
```

If you're filtering or sorting by a column regularly, add an index. It's one of the cheapest performance wins you can get.

## Transactions Prevent Half-Finished Operations

When the automated scheduler runs, it creates dozens of assignments at once. If something fails halfway through — say, a constraint violation on assignment 15 of 30 — you don't want the first 14 to be saved while the rest fail. That leaves your data in an inconsistent state.

Wrapping everything in a transaction means it's all-or-nothing:

```python
try:
    for assignment in new_assignments:
        db.add(assignment)
    db.commit()
except Exception:
    db.rollback()
    raise
```

Either all assignments are saved, or none are. Your data stays consistent.

## The Takeaway

PostgreSQL rewards you for thinking about your data model upfront. Spend an hour sketching out your tables and relationships before writing code. Use foreign keys. Add indexes on columns you query frequently. And always check what SQL your ORM is actually running — the answer might surprise you.
