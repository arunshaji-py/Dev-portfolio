Structuring a React App When You Only Know the Basics

Component folders, state management, API calls — how I organised my first real React project after moving past tutorial-land.

Tutorials teach you how to build a counter and a to-do list. They don't teach you how to organise a React app with multiple dashboards, role-based views, and dozens of API calls. When I started building my staff rostering system's frontend, I had to figure that out on my own.

Here's the structure I landed on and why.

The Problem With Tutorial Structure

Most tutorials put everything in one or two files. That works for learning, but the moment you have an admin dashboard and an employee dashboard with shared components, you need a plan.

My first attempt was a single App.jsx with 800 lines of code. It worked, but finding anything was impossible and every change risked breaking something unrelated.

What I Ended Up With

src/
├── pages/
│   ├── Login.jsx
│   ├── AdminDashboard.jsx
│   └── EmployeeDashboard.jsx
├── components/
│   ├── ShiftCard.jsx
│   ├── AvailabilityCalendar.jsx
│   └── StatsPanel.jsx
├── services/
│   └── api.js
└── utils/
    └── auth.js

The key insight was separating pages (full screens tied to routes) from components (reusable pieces used across pages). The services folder holds all Axios API calls in one place, so I'm not scattering fetch calls throughout components.

Lessons That Stuck

Keep API calls out of components

Early on, I was calling axios.get('/shifts') directly inside component useEffect hooks. This worked but meant the same API URL was hardcoded in five different places. Moving everything to a services/api.js file meant I had one source of truth for every endpoint.

State doesn't always need to be global

I initially thought I needed Redux or Context for everything. Turns out, most state is local to a single component or page. The admin dashboard's shift list doesn't need to be in global state — it's fetched when the page loads and lives in a useState hook.

I only used React Context for two things: the authenticated user and the JWT token. Everything else stayed local.

Conditional rendering gets complex fast

Rendering different UI based on role (admin vs employee), shift status (covered vs uncovered), and time (past vs upcoming) led to deeply nested ternaries that were impossible to read. Breaking these into small, focused components — like a StatusBadge that takes a status prop — made the code dramatically clearer.

What I'd Change

I'd adopt TypeScript from the start. Several bugs I spent hours debugging were simply wrong prop types or mismatched API response shapes. TypeScript would have caught these at write-time instead of run-time.

I'd also look into React Router's data loaders instead of fetching in useEffect. The loading states would be cleaner.

For Other Beginners

Don't wait until your codebase is messy to organise it. Start with a simple folder structure from day one. It doesn't have to be perfect — just having separate folders for pages, components, and API calls will save you hours of confusion later.
